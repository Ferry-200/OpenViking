OpenViking 上传相关调研与讨论总结

  ## 1. 背景

  本次讨论围绕 OpenViking 中“资源上传（Add Resource）”的交互与能力边界展
  开，重点包括：

  - 新增 sidebar 中“添加资源”页面是否需要 Latest Result 面板
  - 原版 console 的 ignore_dirs / include / exclude 等高级选项是否有必要
    保留
  - 原版 path / upload 两种模式的真实含义
  - HTTP 场景下为什么不能直接传本机文件夹路径
  - 现有能力是否支持“上传文件夹”
  - 新版上传 UI 是否需要分成两个 tab

  ———

  ## 2. 当前结论概览

  ### 2.1 关于 Latest Result 面板

  结论：将其从新版 sidebar 中移除是合理的。

  ———

  ### 2.2 关于原版 ignore_dirs / include / exclude

  结论：这些参数本质上是为“目录导入 / 目录扫描”准备的，不是为“单文件上
  传”准备的。

  - ignore_dirs：忽略某些目录
  - include：只包含符合模式的文件
  - exclude：排除符合模式的文件

  这些参数对以下场景有意义：

  - 导入一个目录
  - 扫描一个代码仓库
  - 批量处理目录下的文件集合

  但对于以下场景通常没意义：

  - 单个文件上传
  - 浏览器直接上传一个 PDF / Markdown / TXT

  所以如果新版上传页只做“上传文件”，隐藏这些高级参数是合理的。

  ———

  ## 3. 原版 Add Resource 的真实设计

  原版 console 的 Add Resource 页面实际上复用了一个通用的 add_resource
  请求模型，前端把以下模式混合在一个统一表单里：

  - path 模式
  - upload 模式

  但这两个模式并不是“上传文件”和“上传文件夹”的准确划分。

  ### 3.1 upload 模式

  原版 upload 模式的本质是：

  1. 浏览器上传一个本地文件到服务端临时目录
  2. 服务端返回 temp_file_id
  3. 再调用 /resources 接口，用 temp_file_id 创建资源

  所以它是：单文件上传

  ———

  ### 3.2 path 模式

  原版 UI 文案里写了类似“Path on OpenViking server”，这会让人误以为可以
  直接填：

  - 本机文件路径
  - 本机目录路径
  - 服务端宿主机路径

  但实际上在 HTTP 场景下，不允许这样做。

  HTTP 接口对 path 的约束是：

  - 允许远程资源地址
  - 允许 temp upload 产生的 temp_file_id
  - 不允许直接传宿主机本地路径

  也就是说，原版 path 更接近：

  - 远程 URL
  - 仓库地址
  - 可抓取的网络资源地址

  而不是“随便填写服务器本地文件夹路径”。

  ———

  ## 4. 什么叫“目录导入 / 目录扫描”

  这是我们讨论中最容易混淆的一点。

  ### 4.1 你的原始理解

  你原来理解的“目录导入”是：

  > 给系统一个本机文件夹路径，然后把该目录下所有文件挨个上传

  这个理解在“本地 SDK / 本地进程调用”语境下，其实是接近真实能力的。

  ———

  ### 4.2 在 OpenViking 中更准确的含义

  “目录导入 / 目录扫描”指的是：

  > 当系统拿到一个本地目录路径后，递归遍历目录树，对目录中的文件进行分
  > 类、过滤、解析，再导入资源系统。

  它不只是“挨个上传文件”，而是会做这些事：

  - 遍历整个目录树
  - 跳过某些目录（如 .git、node_modules 等）
  - 应用 ignore_dirs / include / exclude
  - 判断哪些文件可处理、哪些不支持
  - 对可处理文件调用合适解析器
  - 决定是否保留目录结构（preserve_structure）

  所以，“目录扫描”更像是：

  > 一次批量导入前的目录遍历 + 文件筛选 + 解析决策过程

  ———

  ## 5. 为什么 HTTP 场景下 path 不允许传电脑本地文件夹

  这是一个安全边界问题。

  ### 5.1 HTTP 服务和浏览器不是同一台机器的默认假设

  在 HTTP 场景中，前端页面只是客户端，服务端才是真正执行读取文件的地方。

  如果浏览器通过 HTTP 直接传：

  - C:\Users\xxx\Desktop\资料
  - /home/user/docs
  - /var/data/project

  那等价于要求服务器去读它自己的宿主机磁盘路径。

  这会带来几个严重问题：

  - 任意读取服务器本地文件
  - 越权访问宿主机目录
  - 泄露敏感数据
  - 把“前端输入框”变成“服务器文件系统入口”

  所以 HTTP 服务明确禁止这种行为。

  ———

  ### 5.2 HTTP 模式允许什么

  HTTP 模式下，path 允许的是：

  - http://...
  - https://...
  - git@...
  - ssh://...
  - git://...

  以及：

  - 通过 /resources/temp_upload 上传后得到的 temp_file_id

  所以 HTTP 资源来源本质上是两种：

  1. 远程可获取资源
  2. 已经安全进入服务端临时区的上传文件

  ———

  ## 6. 远程资源怎么还会涉及“目录扫描”

  这是第二个关键问题。

  你提到：

  > 远程资源比如 http 这种基本都是单个文件，那还怎么目录扫描？

  这个疑问非常合理。答案是：远程资源不一定只是单文件。

  ### 6.1 远程资源可能是单个文件

  例如：

  - PDF 链接
  - Markdown 链接
  - 文本文件链接
  - 网页下载链接

  这种情况下，系统通常会：

  - 先下载文件
  - 再按文件类型解析

  此时一般不涉及目录扫描。

  ———

  ### 6.2 远程资源也可能是“仓库 / 代码库”

  例如：

  - GitHub 仓库地址
  - git@... 仓库地址
  - ssh://... 仓库地址
  - git://... 仓库地址

  这种情况下，系统会先：

  - git clone
  - 或下载仓库 zip
  - 或拉取 GitHub archive zip

  拿到本地目录后，再对这个目录做扫描和导入。

  所以“目录扫描”在远程资源场景下，常见于：

  - 代码仓库
  - 下载后展开为目录的资源集合
  - 多文件结构化内容

  ———

  ## 7. 它怎么获取远程资源，下载吗？

  结论：是的，本质上就是下载或拉取。

  ### 7.1 对网页 / 文件链接

  系统会：

  - 发起 HTTP 请求
  - 下载远程内容到临时文件
  - 再根据类型走不同解析器

  例如：

  - 下载 PDF 后交给 PDF 解析器
  - 下载 Markdown / Text 后交给文本或 Markdown 解析器
  - 下载 HTML 后本地解析

  ———

  ### 7.2 对代码仓库

  系统会按资源类型走更适合的方式，例如：

  - 对 GitHub 仓库，优先用 ZIP 下载
  - 对其他 repo URL，可能使用 git clone
  - 对 .zip 类型仓库资源，也会先处理归档内容

  因此，”远程获取资源”的本质流程一般是：

  - 下载
  - 或 clone
  - 再 解析
  - 再 写入资源系统

  ### 7.3 Git 仓库的完整处理流程（代码级确认）

  通过阅读后端源码，确认 git 仓库 URL 的处理是**两阶段式**的，而非直接
  clone 到 VikingFS：

  **阶段一：Clone 到服务器本地 OS 临时目录**

  ```
  CodeRepositoryParser.parse()
    → tempfile.mkdtemp(prefix=”ov_repo_”)
    → /tmp/ov_repo_XXXXX/
  ```

  Clone 方式根据 URL 类型选择：
  - GitHub URL → 优先用 GitHub Archive ZIP API 下载 + 解压（更快，无 git 历史）
  - SSH URL (git@) → `git clone --depth 1`
  - 其他 git URL → `git clone --depth 1 --recursive`

  **阶段二：遍历本地临时目录，逐文件上传到 VikingFS 临时区**

  ```
  upload_directory()
    → os.walk(local_dir)
    → 应用 ignore_dirs / include / exclude 过滤
    → 跳过二进制、空文件、>10MB 文件
    → 8 并发上传到 viking://temp/{timestamp}_{uuid}/
  ```

  过滤细节（`upload_utils.py`）：
  - `ignore_dirs`：内置默认跳过 .git、node_modules、__pycache__、venv 等，
    用户可追加，支持目录名和相对路径两种匹配
  - `include`：glob 模式匹配文件名，无模式 = 全部包含
  - `exclude`：glob 模式匹配文件名，或以 `/` 结尾的路径前缀排除
  - 额外跳过：隐藏文件、符号链接、已知二进制扩展名（.exe .dll .pyc .so 等）
  - 文本编码：自动检测并转换为 UTF-8

  **阶段三：TreeBuilder 定稿**

  ```
  TreeBuilder.finalize_from_temp()
    → 从 viking://temp/... 移动到 viking://resources/...
    → GitHub/GitLab 仓库自动提取 org/repo 作为最终目录名
    → 删除临时区
  ```

  **阶段四：清理**

  ```
  finally:
    shutil.rmtree(temp_local_dir)  # 立即删除本地 OS 临时目录
  ```

  **完整数据流**

  ```
  git repo URL
    → CodeRepositoryParser
      → clone 到 /tmp/ov_repo_xxx/          (本地 OS 临时目录)
      → os.walk() 遍历 + 过滤
      → 逐文件上传到 viking://temp/...      (VikingFS 临时区)
      → 删除 /tmp/ov_repo_xxx/
    → TreeBuilder
      → 从 viking://temp/... 移动到 viking://resources/...
    → 语义处理（异步）
      → 生成摘要、向量嵌入、索引
  ```

  ———

  ## 8. 当前 HTTP 上传能力的真实边界

  ### 8.1 目前支持的能力

  当前 HTTP 链路明确支持：

  - 上传单个文件
  - 获取 temp_file_id
  - 用 temp_file_id 创建资源

  所以浏览器上传层面当前是：

  > 单文件上传能力

  ———

  当前没有看到 HTTP 层原生支持：

  - 浏览器直接上传文件夹
  - 多文件 + 相对路径上传
  - 返回 temp_dir_id
  - 直接把上传目录交给后端做目录扫描

  > 一个上传组件，用户可直接选择文件夹，后端自动递归导入

  当前现有 HTTP 接口还不支持完整落地。

  ———

  ## 9. 对新版上传 UI 的建议

### 9.1 当前结论：不支持真正的文件夹上传链路

  虽然系统后端具备目录处理能力，但当前 HTTP 上传链路只支持：

  - 单文件 temp_upload
  - 返回 temp_file_id
  - 再调用 /resources

  缺少以下关键能力：

  - 多文件上传并保留相对路径
  - 上传整个目录并返回 temp_dir_id
  - 基于 temp_dir_id 执行目录扫描和导入

  ### 9.2 这意味着什么

  当前浏览器侧不能直接做到：

  > 选择一个文件夹 → 上传整个文件夹 → 服务端按目录结构导入资源

  所以如果现在在 UI 上直接做“上传文件夹”，但后端不支持，就会造成体验和能
  力不一致。

  ———

  ## 10. 关于 .zip 能不能替代“上传文件夹”

  ### 10.1 当前不能简单等价

  虽然系统里存在 .zip 解析能力，但当前 .zip 更偏向于：

  - 读取压缩包内容
  - 生成说明性结构
  - 转成 markdown 或描述性结果

  它不等价于：

  - 把 ZIP 当作目录原样展开
  - 然后把其中的每个文件按目录导入资源系统

  ### 10.2 结论

  当前不能简单把：

  > “上传文件夹”

  替换为：

  > “让用户手动压成 zip 上传”

  至少从现有能力角度看，这两者不完全等价。

  ———

  ## 11. 新版资源上传 UI 的建议

  ### 11.1 当前阶段推荐方案

  如果当前后端只支持单文件上传，那么新版 sidebar 最合理的方案是：

  - 只保留“上传文件”
  - 不暴露目录导入高级参数
  - 不做 path / upload 两个 Tab
  - 用更面向业务的文案，而不是暴露底层接口概念

  ### 11.2 如果未来要支持“文件夹上传”

  建议不要做成旧式的 path / upload Tab，而是做成：

  - 一个统一的“上传资源”组件
  - 两个操作入口：
      - 选择文件
      - 选择文件夹
  - 如果用户选的是文件夹：
      - 再展开高级选项
      - 如 ignore_dirs/include/exclude/preserve_structure

  这种方式更符合用户心智。

  ———

  ## 12. 是否有必要分成两个 Tab

  ### 12.1 当前判断

  不太有必要。

  原因：

  - “上传文件”和“上传文件夹”本质上属于同一个任务：导入资源
  - 用户更关心“我要导入什么”，而不是底层调用的是哪条 API
  - Tab 方式容易把实现细节暴露给用户

  ### 12.2 更合理的交互

  更推荐：

  - 一个统一上传入口
  - 根据用户选择的对象类型动态调整 UI
  - 普通文件：简洁模式
  - 文件夹：高级模式

  不过这一方案的前提是：

  > 后端必须真正支持目录上传链路

  当前还不具备这一条件。

  ———

  ## 13. 如果后续要真正支持”上传文件夹”，需要补什么能力

  ### 13.1 关键认知：后端的目录处理管道已经存在

  通过源码调研确认，后端处理 git 仓库时的流程本质上是：

  1. 把远程内容拉到**服务器本地临时目录**（/tmp/ov_repo_xxx/）
  2. 对这个临时目录做 `os.walk()` **遍历 + 过滤 + 逐文件上传**到 VikingFS

  这意味着，从”拿到一个本地临时目录”开始，后面的整套管道（遍历、过滤、
  上传、TreeBuilder 定稿）已经完整实现了。

  浏览器上传文件夹和 git clone 的区别**仅在于第一步**——内容从哪来：

  | 来源 | 第一步 | 第二步（完全相同） |
  |------|--------|---------------------|
  | git URL | git clone 到 /tmp/ | os.walk() → 过滤 → 上传到 VikingFS |
  | 浏览器文件夹 | multipart 上传到 /tmp/ | os.walk() → 过滤 → 上传到 VikingFS |

  所以”上传文件夹”不需要让后端从零实现目录处理——只需要一个入口把浏览器
  的文件送进那个临时目录，后面的管道全部复用。

  ### 13.2 后端实际需要补充的

  只需要新增一个接口入口：

  ```
  POST /api/v1/resources/temp_upload_dir    ← 目前不存在
  ```

  这个接口做的事情：
  - 接收多个文件 + 每个文件的相对路径（如 src/main.py、docs/readme.md）
  - 在服务端临时目录里按相对路径还原目录结构
  - 返回 `temp_dir_id`（对应现有的 `temp_file_id` 概念）

  对比现有接口：

  | 接口 | 输入 | 返回 | 状态 |
  |------|------|------|------|
  | `POST /resources/temp_upload` | 单个文件 | `temp_file_id` | 已有 |
  | `POST /resources/temp_upload_dir` | 多文件 + 相对路径 | `temp_dir_id` | 需新增 |

  ### 13.3 /resources 需要新增目录输入

  当前 /resources 主要接受：

  - path
  - temp_file_id

  后续应扩展为也能接受：

  - temp_dir_id

  然后在 temp_dir_id 场景下才启用：

  - ignore_dirs
  - include
  - exclude
  - preserve_structure

  ### 13.4 复用程度评估

  如果要实现”浏览器上传文件夹”，需要新写的代码量很小：
  - 新增 `temp_upload_dir` 接口（接收多文件 + 相对路径，写入临时目录）
  - `/resources` 新增对 `temp_dir_id` 的支持

  可以直接复用的已有代码：
  - `upload_directory()`（目录遍历 + 过滤 + 并发上传到 VikingFS）
  - `ignore_dirs` / `include` / `exclude` 过滤逻辑
  - `TreeBuilder.finalize_from_temp()`（从临时区移动到最终位置）
  - 所有文件解析器（PDF、代码、图片等）

  **结论：架构上差距比表面看起来小得多，核心管道已经全部存在。**

  ———

  ## 14. 最终结论

  ### 14.1 结论一：原版那些高级参数不是单文件上传必须的

  它们主要是为目录扫描 / 目录导入准备的。

  ### 14.2 结论二：HTTP 模式下 path 不是“你电脑上的本地路径”

  HTTP 模式下，path 只允许远程资源地址，不允许直接传本地文件夹路径。

  ### 14.3 结论三：当前系统有目录处理能力，但没有浏览器侧完整目录上传能
  力

  也就是说：

  - 能处理目录，不等于能从浏览器直接上传目录
  - 但后端从"拿到临时目录"到"导入资源系统"的完整管道已经存在
    （git clone 就在用这套管道）
  - 缺的只是一个让浏览器把文件夹内容送进临时目录的入口接口
    （temp_upload_dir → temp_dir_id）

  ### 14.4 结论四：当前新版 sidebar 最佳实践

  - 先只做单文件上传
  - 不暴露目录高级参数
  - 不做 path / upload 双 Tab
  - 成功提示用轻量反馈即可

  ### 14.5 结论五：未来如果要支持文件夹上传

  推荐做成：

  - 一个统一上传组件
  - 文件 / 文件夹两种选择
  - 文件夹时才显示高级选项

  但这需要后端补齐目录上传链路。

  ———

  ## 15. 建议的产品决策

  ### 短期建议

  - 新版资源上传页面只保留“上传文件”
  - 保持简单、明确、低认知成本

  ### 中期建议

  - 如果业务确实需要“上传文件夹”，先设计后端目录上传协议
  - 不建议直接套用原版 path / upload 交互

  ### 长期建议

  - 形成统一“资源导入”模型：
      - 单文件
      - 远程 URL
      - 仓库地址
      - 文件夹上传
  - 由 UI 根据资源类型动态展示参数，而不是一开始就把所有字段堆给用户

  ———
